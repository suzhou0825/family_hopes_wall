# 家庭许愿墙开发约束

## 沟通与文档

- 使用简体中文沟通。
- 复杂任务先讨论思路，再实施。
- 回答要精炼，避免重复和空泛表述。
- 每次产品或代码更新，都必须同步更新：
  - `docs/USER_GUIDE.md`
  - `docs/DEVELOPMENT_CONSTRAINTS.md`
  - `docs/PRODUCT_PLAN.md`
  - `docs/CHANGELOG_NOTES.md`
  - 涉及部署时同步更新 `docs/DEPLOYMENT.md`
- 用户提供重要信息时，需要提醒安全隐患，不要继续暴露类似信息。

## 产品边界

- 当前先做 Web MVP，不做苹果安装包。
- 本地开发端口固定为 `3333`。
- 不使用 `3000` 作为默认端口。
- 不使用 `6666`，该端口会被浏览器拦截。
- 外网部署优先使用 Vercel，部署说明维护在 `docs/DEPLOYMENT.md`。
- 当前版本不做支付、商城、物流、真实采购。
- 兑现由家庭线下完成，系统只记录状态。

## 技术约束

- 技术栈：Next.js + TypeScript。
- 云端方向：Supabase。
- 已接入 Supabase Auth 和 `app_state` 数据表。
- 当前 Supabase 项目已执行 `supabase/schema.sql`。
- 当前 Supabase 第一版使用 `app_state.data` JSON 保存完整应用数据。
- 后续需要拆成家庭成员、愿望、任务、打卡记录等独立数据表。
- 未配置或未登录 Supabase 时，使用浏览器 `localStorage` 作为演示数据。
- 修改数据模型时必须兼容旧的 localStorage 数据，避免用户本地数据直接崩溃。
- 不允许把 Supabase `service_role` key 放进前端代码，只能使用 anon key。
- `.env.local` 可以保存当前项目的 Supabase URL 和 anon/publishable key，但不得提交到版本库。
- 支持 GitHub OAuth 登录，但 GitHub Client Secret 只能配置在 Supabase 后台，不能写进前端代码或 `.env.local`。
- 新增交互后需要跑：
  - `npm run typecheck`
  - `npm run build`
- 重要界面变更需要用浏览器验证。

## 角色与权限

- 父母账号可以访问任务发布和家庭管理。
- 孩子账号可以访问许愿板块、我的任务，并在许愿墙申领任务。
- 父母账号不显示孩子许愿入口。
- 孩子账号不显示任务发布入口。
- 家庭成员由管理员手动添加，不使用邀请码。

## 家庭成员

- 添加或编辑父母时，设置爸爸或妈妈身份。
- 添加或编辑孩子时，不显示父母身份。
- 添加或编辑孩子时，必须支持选择性别：男孩、女孩。
- 添加或编辑孩子时，必须支持选择家庭称呼：哥哥、弟弟、姐姐、妹妹。

## 愿望规则

- 孩子可以发布、编辑、删除愿望。
- 愿望必须支持：
  - 愿望名称
  - 愿望描述
  - 愿望类型
  - 期望兑现人
  - 期望时间
- 愿望类型包括：物质奖励、旅游奖励、陪玩奖励、其他愿望。
- 愿望状态为：待申领、兑换中、兑现中。

## 任务规则

- 父母可以发布、编辑、删除任务。
- 任务类型包括：打卡任务、一次性任务、承诺任务。
- 任务奖励必须支持：
  - 奖励类型
  - 奖励描述
  - 可选关联愿望
- 奖励类型必须与愿望类型保持一致。
- 任务状态为：待申领、完成中、已完成。
- 任务完成必须由父母审核通过。
- 孩子可以在许愿墙主动申领任务。
- 孩子需要有我的任务板块，展示自己申领的任务和进展。

## 打卡任务

- 只有任务类型为打卡任务时，才显示打卡频率和截止时间。
- 打卡频率按周一到周日让父母多选。
- 打卡任务必须设置截止时间。
- 孩子可以按天打卡。
- 任务看板和我的任务需要展示打卡进度。

## UI 约束

- 界面需要适配手机、iPad 和桌面浏览器。
- 主要页面不能出现横向溢出。
- 表单字段只在相关场景展示，避免无关字段干扰。
- 父母视角和孩子视角必须清晰区分。
