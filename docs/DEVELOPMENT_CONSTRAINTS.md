# 家庭许愿墙开发约束

## 沟通与文档

- 使用简体中文沟通。
- 复杂任务先讨论思路，再实施。
- 回答要精炼，避免重复和空泛表述。
- 每次产品或代码更新，都必须同步更新：
  - `docs/USER_GUIDE.md`
  - `docs/DEVELOPMENT_CONSTRAINTS.md`
  - `docs/PRODUCT_PLAN.md`
  - `docs/CHANGELOG_NOTES.md`
  - 涉及部署时同步更新 `docs/DEPLOYMENT.md`
- 用户提供重要信息时，需要提醒安全隐患，不要继续暴露类似信息。
- 项目交接文档固定维护在 `handover/PROJECT_HANDOVER.md`。
- 交接文档生成规则固定维护在根目录 `PROJECT_HANDOVER_GENERATION_SPEC.md`。
- 新窗口接手提示固定维护在根目录 `NEW_AGENT_HANDOFF_PROMPT.md`。
- 更新交接文档前必须将旧版备份到 `handover/archive/PROJECT_HANDOVER_YYYY-MM-DD_HH-mm.md`。
- 交接文档必须以当前代码和 Git 状态为准，未确认的平台信息必须标为待确认。
- 交接文档不得包含真实密码、会话 token、`service_role` 或 OAuth Secret。

## 产品边界

- 当前先做 Web MVP，不做苹果安装包。
- 本地开发端口固定为 `3333`。
- 不使用 `3000` 作为默认端口。
- 不使用 `6666`，该端口会被浏览器拦截。
- 外网部署优先使用 Vercel，部署说明维护在 `docs/DEPLOYMENT.md`。
- 当前版本不做支付、商城、物流、真实采购。
- 兑现由家庭线下完成，系统只记录状态。

## 技术约束

- 技术栈：Next.js + TypeScript。
- 云端方向：Supabase。
- 已接入应用自有账号体系，不使用 Supabase Auth 作为家庭成员登录。
- 家庭、账号、密码哈希、会话和家庭数据分别保存到 `app_families`、`app_accounts`、`app_sessions`、`app_family_data`。
- 积分账户、积分流水、宠物领养和宠物互动必须分别保存到 `app_point_accounts`、`app_point_transactions`、`app_pet_adoptions`、`app_pet_interactions`，不得写入本地演示状态。
- 旧版本遗留的 `profiles`、`app_state`、`app_data` 必须从 Supabase 中删除，不再保留冗余表。
- 前端只能使用 Supabase anon key 调用 RPC，不得直接读取密码哈希或会话表。
- Supabase 项目需要执行最新的 `supabase/schema.sql`。
- 本地 `http://localhost:3333` 调试环境、GitHub 代码和 Vercel 线上环境必须使用同一个 Supabase 项目和数据库。
- 本地 `.env.local` 与 Vercel Environment Variables 中的 `NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY` 必须指向同一个 Supabase 项目。
- 愿望、任务和家庭成员仍使用 `app_family_data.data` JSON；积分和宠物数据已拆为独立表。
- 后续需要拆成家庭成员、愿望、任务、打卡记录等独立数据表。
- 未配置 Supabase 时，使用浏览器 `localStorage` 作为开发兜底。
- 修改数据模型时必须兼容旧的 localStorage 数据，避免用户本地数据直接崩溃。
- 不允许把 Supabase `service_role` key 放进前端代码，只能使用 anon key。
- `.env.local` 可以保存当前项目的 Supabase URL 和 anon/publishable key，但不得提交到版本库。
- 登录状态检查必须有超时或未配置兜底，不能让用户无限停留在加载页。
- 新增交互后需要跑：
  - `npm run typecheck`
  - `npm run build`
- 重要界面变更需要用浏览器验证。

## 角色与权限

- 打开网页后必须先登录账号密码，未登录不得进入应用主界面。
- 登录账号由本应用管理，登录时通过 Supabase RPC 校验数据库中的账号密码哈希。
- 账号资料必须记录到 Supabase `app_accounts` 表。
- 动画头像只能使用系统允许的头像编号，编号保存到 `app_accounts.avatar_id`，不得仅保存在浏览器或家庭 JSON。
- 头像编号配置统一维护在 `lib/avatar-config.ts`，当前允许 `father_01`、`mother_01`、`older_brother_01`、`younger_brother_01`、`older_sister_01`、`younger_sister_01`。
- 注册只支持普通账号，不使用邮箱作为登录账号。
- 主动注册只能创建父母账号。
- 主动注册会创建新的家庭数据空间。
- 父母注册时必须选择爸爸或妈妈。
- 第二个父母账号必须由已登录父母在家庭管理中创建，才能加入同一个家庭。
- 父母可以在家庭管理中创建父母账号和孩子账号。
- 父母创建孩子账号时必须设置孩子账号、初始密码、性别和家庭称呼。
- 账号只能包含小写字母、数字、下划线，长度 3 到 32 位。
- 注册密码必须输入两次并校验一致。
- 密码至少 8 位，并且必须包含字母和数字。
- 密码中字母区分大小写。
- 登录、注册、修改密码和父母创建成员账号的密码框必须使用眼睛图标切换显示/隐藏，不显示文字按钮。
- 登录页不得显示密码规则提醒文字；注册和修改密码仍必须执行实际校验。
- 账号管理修改密码必须填写确认新密码，两次输入一致后才允许提交。
- 父母账号可以访问任务发布和家庭管理。
- 孩子账号可以访问许愿板块、我的任务，并在许愿墙申领任务。
- 父母账号不显示孩子许愿入口。
- 孩子账号不显示任务发布入口。
- 登录后右上角必须只读展示当前登录账号绑定的家庭成员和具体角色。
- 不允许通过前端切换成其他家庭成员身份；角色权限必须以登录账号的 `member_id` 和 `role` 为准。
- 家庭成员由父母手动添加，不使用邀请码。

## 家庭成员

- 父母身份在主动注册时设置为爸爸或妈妈。
- 家庭管理负责添加父母账号、孩子账号和编辑家庭成员资料。
- 删除家庭成员必须同步删除对应 `app_accounts` 账号和 `app_sessions` 会话。
- 不能删除当前正在登录的账号。
- 添加或编辑孩子时，必须支持选择性别：男孩、女孩。
- 添加或编辑孩子时，必须支持选择家庭称呼：哥哥、弟弟、姐姐、妹妹。

## 愿望规则

- 孩子可以发布、编辑、删除愿望。
- 愿望必须支持：
  - 愿望名称
  - 愿望描述
  - 愿望类型
  - 期望兑现人
  - 期望时间
- 愿望类型包括：物质奖励、旅游奖励、陪玩奖励、积分奖励、其他愿望。
- 愿望必须记录提交时间并在许愿墙展示；旧数据没有时间时显示历史数据未记录。
- 愿望新增或编辑时，名称不得与任何未归档愿望完全一致，比较时忽略首尾空格和大小写。
- 愿望状态为：待申领、兑换中、兑现中、已兑换。
- 只有父母可以将兑现中的愿望确认成已兑换；已兑换愿望从许愿墙隐藏并保留在历史愿望中。

## 任务规则

- 父母可以发布、编辑、删除任务。
- 任务类型包括：打卡任务、一次性任务、承诺任务。
- 任务奖励必须支持：
  - 奖励类型
  - 奖励描述
  - 可选关联愿望
- 奖励类型必须与愿望类型保持一致。
- 孩子申报任务或愿望选择成长星奖励时，只允许 1 到 100；选择非积分奖励时不得附加成长星。
- 父母发布任务可以使用纯成长星奖励，也可以在非积分奖励基础上附加成长星。
- 任务成长星只能在父母审核完成时通过 RPC 入账，并以任务编号防止重复发放。
- 孩子申报任务必须选择奖励类型，非积分奖励必须填写建议奖励说明。
- 任务必须记录提交时间并在任务公示展示；旧数据没有时间时显示历史数据未记录。
- 任务新增、编辑或申报时，名称不得与任何未完成任务完全一致，比较时忽略首尾空格和大小写。
- 父母复制任务时保留类型、奖励和打卡频率，清除领取人、打卡、提交状态和关联愿望；名称自动追加副本编号，打卡截止时间按原周期顺延。
- 任务状态为：待申领、完成中、已完成。
- 已完成任务不显示在许愿墙任务看板，但必须保留在孩子的“我的任务”和父母历史任务中。
- 我的任务必须展示任务提交时间；孩子提交完成时记录 `completionSubmittedAt`，父母审核通过时记录 `completedAt`。
- 旧任务缺少完成时间时必须显示历史数据未记录，不得使用创建时间代替。
- 任务完成必须由父母审核通过。
- 孩子可以在许愿墙主动申领任务。
- 孩子可以申报新任务，申报任务在父母审批前不得进入任务公示或可申领列表。
- 父母审批通过孩子申报的任务后，任务进入待申领状态，孩子可以自行申领。
- 孩子需要有我的任务板块，展示自己申领的任务和进展。

## 打卡任务

- 只有任务类型为打卡任务时，才显示打卡频率和截止时间。
- 打卡频率按周一到周日让父母多选。
- 打卡任务必须设置截止时间。
- 孩子可以按天打卡。
- 任务看板和我的任务需要展示打卡进度。

## UI 约束

- 界面需要适配手机、iPad 和桌面浏览器。
- 主要页面不能出现横向溢出。
- 表单字段只在相关场景展示，避免无关字段干扰。
- 父母视角和孩子视角必须清晰区分。
- 登录页与应用整体保持柔和粉紫配色、较大圆角和轻量立体阴影；登录插画必须使用原创或已获授权资源。
- 登录页不得因视觉参考恢复邮箱登录、Supabase Auth 或第三方 OAuth 入口。
- 登录后首页必须采用许愿墙、任务公示、积分区三栏入口；窄屏按单列排列。
- 首页不显示额外导航条，三张主卡片整体可点击；专属页面提供返回首页入口。
- 首页三张卡片标题靠左显示，不显示 01、02、03 等序号，标题使用统一的展示字体。
- 顶部账号区不得展示 Supabase 保存状态等内部技术信息。
- 家庭成员卡片必须展示头像；成员缺少头像编号时按角色显示默认头像。
- 许愿和任务卡片必须动态展示现有未完成数据，无数据时显示动态空状态。
- 首页许愿和任务滚动条目必须展示提交日期和时间；旧数据不得使用创建日期伪造精确提交时间。
- 许愿墙和任务公示使用固定高度滚动窗口，一次完整展示约 4 条内容，不得随数据数量无限增高。
- 许愿墙和任务公示摘要位于卡片底部上方，并保留适当上下留白，不得挤压滚动窗口。
- 发布愿望按钮只对孩子展示，发布任务按钮只对父母展示。
- 发布愿望、发布任务和申报任务入口必须位于卡片头部。
- 成长星账户初始余额为系统授予 100，并必须生成对应积分流水。
- 所有积分增减、实物兑换、虚拟兑换、宠物领养和弃养必须通过安全 RPC 原子执行，前端不得直接修改余额。
- 电子宠物库固定使用配置编号；领取关系保存到独立数据表。只有孩子能领取，每个孩子最多 2 只，父母只能查看状态。
- 第一只宠物免费，第二只扣 500 成长星；弃养扣 2000 成长星，余额不足不得执行。
- 宠物素材必须使用原创或已获授权资源，不得使用受版权保护的角色形象。
- 首页宠物陈列区必须充分使用积分卡剩余空间，不得只显示小尺寸缩略图；两只宠物使用自动轮播，不能撑高首页。
- 宠物成长值由领养日期确定，当前规则为领养当天第 1 天、每天累计 10 点；每日心情与感悟必须按宠物和日期稳定生成，不能因页面刷新随机变化。
- 宠物动作必须切换实际姿态动作帧，不能只平移或旋转整张图片；动作按不定时间隔触发，并遵循减少动态效果的系统设置。
- 宠物交互页第一阶段支持喂养、玩耍和装扮，所有状态和互动记录必须保存到 Supabase；拍照分享和串门暂不开发。
