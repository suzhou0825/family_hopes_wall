# 产品决策记录

## 2026-06-14

- 产品名称确定为家庭许愿墙。
- 先做 Web MVP，不生成苹果安装包。
- 应用需要优先适配手机、iPad 和桌面浏览器。
- 使用云端同步作为后续方向。
- 技术栈默认使用 Next.js + TypeScript + Supabase。
- 初始版本先用前端模拟数据跑通流程，再接 Supabase。
- 家庭成员由管理员手动添加，不使用邀请码。
- 任务完成必须由父母审核通过。
- 愿望状态为待申领、兑换中、兑现中、已兑换。
- 任务状态为待申领、完成中、已完成。
- 已兑换愿望和已完成任务采用逻辑归档，不直接删除业务数据。
- 家庭角色由登录账号绑定，不允许在前端切换为其他家庭成员。
- 登录后首页采用许愿墙、任务公示、积分区三栏导航布局。
- 头像使用内置动画头像库，头像编号保存在 `app_accounts.avatar_id`。
- 内置头像库按爸爸、妈妈、哥哥、弟弟、姐姐、妹妹六类家人角色维护。
- 孩子申报的任务必须先由父母审批，审批通过后才进入公示和申领流程。
- 顶部账号区不展示数据库保存状态等内部技术信息。
- 修改密码必须二次输入并校验一致。
- 未归档愿望和未完成任务不允许名称重复。
- 积分奖励已驱动真实成长星账户，父母审核任务时通过安全 RPC 入账并防止重复发放。
- 复制任务不复制领取、执行和愿望关联状态。
- 积分账户、流水、兑换和宠物业务使用 Supabase 独立表；孩子初始由系统授予 100 成长星。
- 第一只宠物免费，第二只 500 成长星，弃养扣 2000 成长星；每个孩子最多 2 只。
- 奖品目录和兑换记录使用独立 Supabase 表；父母维护实物或虚拟奖品，孩子通过 RPC 完成真实兑换。
- 宠物动作采用配置化姿态帧和受控动作状态，主动互动优先于随机空闲动作；装扮使用按宠物校准的 SVG 覆盖层。
